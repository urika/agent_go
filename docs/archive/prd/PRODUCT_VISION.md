# 产品核心定位：一句话讲清楚 agent_go 是什么、给谁用、为什么选它

> 版本: v1.0
> 日期: 2026-07-24
> 定位: 产品核心价值主张，所有 PRD/SPEC/NFR 的北极星

---

## 一、核心用户

**每周用 Claude Code 超过 20 次的工程师** — 他们已经信任 AI 写代码，但厌倦了"一个任务拆成五次对话、每次手动传递上下文、逐个验证结果"的机械劳动。

这不是一个"能编程的聊天机器人"的目标用户。这是**已经把聊天机器人当工具、现在需要批处理能力**的用户。

---

## 二、一句话目标场景

> **你说一遍任务，关掉终端。回来时，拆解好的子任务在隔离环境里执行完毕，代码已提交，验证已通过，PR 已生成。**

---

## 三、一句话核心价值

> **agent_go 让 Claude Code 从"对话式结对编程"升级为"异步任务委派"——你说需求，它交 PR。**

---

## 四、用户选择 agent_go 的三个理由

对"一次 Claude Code 对话"而言：

| | Claude Code (裸用) | agent_go |
|---|---|---|
| 多步骤任务 | 需要人工拆分、逐个执行、手动传递上下文 | 一次输入，自动 Plan → Decompose → Execute |
| 执行过程 | 你必须盯着屏幕，交互确认每一步 | 无头模式全程自主，你只需要看结果 |
| 产物 | 文件被修改了，commit message 自己写 | Conventional Commits + 验证报告 + PR 模板 |
| 安全 | 你手动确认每个操作 | 验证命令白名单 + 沙箱环境 + 审计日志 |
| 规模 | 一个任务 = 一次对话 | 一个任务 = 3-5 个子任务，可并发 |

**差异化就是一句话：Claude Code 是交互工具，agent_go 是异步执行引擎。**

---

## 五、所有 NFR 归到一句话测试

**"你敢不敢在下午 4 点输入 `agent_go run`，关电脑走人，周一早上信心满满地 merge PR？"**

如果对以下任何一个问题的答案是"不放心"，产品就没达到目标：

| 不放心的点 | 对应的 NFR KPI | 达标判据 |
|-----------|---------------|----------|
| 它能跑完吗？不会卡在半路？ | 任务成功率 | ≥ 97% 能独立完成 |
| 它不会把我的密钥泄露出去吧？ | 安全零事故 | 0 次凭证泄漏 / 白名单绕过 |
| 我的 API 账单会不会爆炸？ | 单任务成本 | ≤ $0.03/次 |
| 它改的代码我敢合吗？ | 验证通过率 | 所有 subtask 的 verify_ok=True |
| 它改了什么我能查吗？ | 可观测性 | 每次变更 = 文件列表 + diff stat + 验证结果 |

---

## 六、非目标用户（有意识放弃）

| 用户 | 放弃原因 |
|------|----------|
| 不用 Claude Code 的开发者 | agent_go 是 Claude Code 的编排层，不是替代品 |
| 零技术背景的"prompt engineer" | agent_go 的输出是代码变更，用户需要能 review git diff |
| 单步骤任务的用户 | 用裸 Claude Code 更快 — agent_go 的编排开销在单步骤场景下是负资产 |

**不做大而全。只为"需要 AI 自主完成多步骤代码变更"的工程师做最好的编排工具。**
