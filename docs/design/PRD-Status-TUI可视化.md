## 一、背景与问题

### 1.1 现状

`agent_go status --watch` 提供基础轮询刷新，输出表格行到终端。`--verbose` 追加最后 2 条 Claude 事件日志。

### 1.2 核心问题

| 问题 | 影响 |
|------|------|
| 单表无法展示 subtask 级粒度 | 用户不知道每个 subtask 的具体状态 |
| 轮询刷新闪烁 (`os.system("clear")`) | 体验差，无法在刷新间隔内交互 |
| 无实时日志流 | `--verbose` 只显示最后 2 条静态行 |
| 无交互能力 | 无法展开/折叠、切换任务、暂停监控 |
| 多任务并发时信息密度过高 | 5 个任务同时跑，表格不可读 |

## 二、功能需求

### 3.1 F-T1: 多面板 TUI 布局

**布局**:

```
┌─ agent_go Status ──── [q]退出 [j/k]导航 [Enter]展开 [r]刷新 ────┐
│                                                                  │
│ ┌─ 任务列表 ──────────────────────┐ ┌─ 详情面板 ────────────────┐ │
│ │ 🔄 task-xxx 重构认证   2/5 3m   │ │ [sub-3] 编写测试          │ │
│ │    ├─ ✅ sub-1 JWT     1m05s    │ │ Agent: tester             │ │
│ │    ├─ ✅ sub-2 登录    0m52s    │ │ Skill: tdd-workflow       │ │
│ │    ├─ 🔄 sub-3 测试    1m15s    │ │ 变更: 3 files, +45/-2     │ │
│ │    ├─ ⏳ sub-4 文档       -     │ │ 验证: 通过                │ │
│ │    └─ ⏳ sub-5 清理       -     │ │                          │ │
│ │ 🟢 task-yyy 完成             ✅ │ │ ┌─ 实时日志 ────────────┐ │ │
│ │ 🔴 task-zzz 失败             ❌ │ │ │ 15:03:12 [Write] ...  │ │ │
│ └────────────────────────────────┘ │ │ 15:03:14 [Bash] ...   │ │ │
│                                    │ │ 15:03:15 ✅ pass      │ │ │
│                                    │ └──────────────────────┘ │ │
│                                    └──────────────────────────┘ │
│ ┌─ 状态栏 ────────────────────────────────────────────────────┐ │
│ │ 3 任务 | 1 运行中 | API: 2 次, 缓存: 1 次                  │ │
│ └─────────────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────────┘
```

| 面板 | 刷新 | 内容 |
|------|------|------|
| 任务列表（左） | 1s | 树形任务→subtask，折叠/展开 |
| 详情面板（右上） | 1s | 选中 subtask 的 Agent/Skill/变更/验证 |
| 实时日志（右下） | 500ms | tail -f 最后 20 行执行日志 |
| 状态栏（底部） | 1s | 全局统计摘要 |

**颜色编码**: 绿色=完成, 黄色=运行中, 红色=失败, 灰色=待执行, 青色=选中项

### 3.2 F-T2: 键盘交互

| 按键 | 功能 |
|------|------|
| `j`/`↓` | 光标下移 |
| `k`/`↑` | 光标上移 |
| `Enter` | 展开/折叠任务 |
| `r` | 强制刷新 |
| `q`/`Esc` | 退出 |
| `1-4` | 过滤：全部/运行中/已完成/失败 |

### 3.3 F-T3: 实时日志流

- 选中 subtask 后显示其 `execution.log` 尾部 20 行
- 工具调用高亮（青色）、错误（红色）、成功（绿色）
- 自动滚动到最新

### F-T4: 任务过滤、排序与搜索

**优先级**: P1

| 操作 | 按键 | 功能 |
|------|------|------|
| 状态过滤 | `1` `2` `3` `4` | 全部 / 运行中 / 已完成 / 失败 |
| 排序切换 | `s` | 最新优先 / 最旧优先 |
| 搜索 | `/` | 输入关键词，过滤 task_id 或任务描述 |
| 清除过滤 | `Esc` | 恢复默认显示 |

### F-T5: 资源监控

**优先级**: P1

- 状态栏显示活跃 claude 进程数
- API 调用统计（本会话累计 + 缓存命中）
- 进程状态监控（基于 `active_pids` 或 `ps`）

### F-T6: 复杂工作流显示

**优先级**: P2（依赖 D-7 工作流表达改进）

支持三种扩展工作流模式的显示：

**循环迭代**: 增加"迭代组"层级，显示迭代计数器 `[2/5]`、终止条件、每轮度量指标，支持折叠/展开每轮。

**条件分支**: 探索阶段（🔍）+ 决策点（🎯）+ 选中分支正常显示，未选分支灰显折叠。

**嵌套子计划（HTN）**: 多层缩进 `.1 .2 .3`，父节点状态由子节点聚合，默认折叠，Enter 展开。

详见第五章"复杂工作流显示设计"。

## 三、信息项清单

### 3.1 任务级字段

| 字段 | 来源 | 说明 |
|------|------|------|
| task_id | meta.task_id | 任务唯一标识 |
| 任务描述 | meta.task | 截断至 40 字符 |
| 状态图标 | 计算 | 彩色 emoji |
| 完成进度 | results 统计 | `2/5` + 进度条 |
| 耗时 | created vs now/log_mtime | 运行中实时，已完成冻结 |
| Issue | meta.issue | `#42` |
| 当前 subtask | 日志解析 | 最近一次 sub-N START 的标题 |
| Agent 类型 | meta.agent_type | 全局默认 Agent |
| 并发 worker 数 | meta | `--parallel` 参数 |

### 3.2 Subtask 级字段（展开后）

| 字段 | 来源 | 说明 |
|------|------|------|
| subtask_id | result.json / subtask | `sub-1` |
| 标题 | subtask.title | 截断至 50 字符 |
| 状态 | result.status | 彩色编码 |
| Agent 类型 | subtask.agent_type / result | `tester` |
| Agent 来源 | result.agent_type_source | `[LLM]` `[规则]` `[默认]` |
| Skills | subtask.skills | 已加载的 Skill 列表 |
| 未命中 Skill | result.skills_unresolved | ⚠️ 标记 |
| 变更摘要 | result.summary | `3 files, +45/-2` |
| 验证结果 | result.verify_ok | `通过` / `失败` / `跳过` |
| 耗时 | result.duration_sec | 格式化 `1m05s` |
| Sandbox 类型 | result.sandbox_type | `headless` / `greywall` / `native` |

### 3.3 详情面板字段（选中 subtask 时）

| 字段 | 说明 |
|------|------|
| Agent Prompt 摘要 | 前 100 字符 |

| 风险提示 | risks 列表 |

### 3.4 实时日志面板

| 内容 | 来源 | 高亮 |
|------|------|------|

### 3.5 全局状态栏

| 字段 | 说明 |
|------|------|

| API 调用次数 | 会话级统计 |
| 缓存命中次数 | Plan 缓存统计 |

## 四、图形展示能力

curses 是纯文本终端，使用 Unicode 字符画模拟图形。

### 4.1 进度条

```
sub-1  ████████████████░░░░  80%  1m05s
sub-2  ██████████░░░░░░░░░░  50%  运行中...
sub-3  ░░░░░░░░░░░░░░░░░░░░   0%  等待
sub-4  ████████████████████ 100%  完成 ✅
```

使用 `█`（U+2588）和 `░`（U+2591）字符组合，宽度 20 字符 = 每格 5%。

### 4.2 依赖图（ASCII DAG）

```
     sub-1 ✅
    ┌───┴───┐
  sub-2 ✅ sub-3 🔄
    └───┬───┘
     sub-4 ⏳
```

使用 `┌─┴─┐` `├─` `└─` 等 box-drawing 字符绘制依赖树。仅在详情面板中显示（空间有限）。

### 4.3 资源面板（迷你仪表盘）

```
Claude 进程:  ████░░ 2/4 workers active
API 调用:     █░░░░░ 3 次 (缓存命中: 2)
```

### 4.4 能力边界

| 能力 | TUI 支持 | 说明 |
|------|---------|------|
| 进度条 | ✅ | Unicode 字符画 |
| 依赖图 | ✅ | 简单 DAG（≤5 节点） |
| 折线图/趋势 | ⚠️ | sparkline（4 行高迷你图） |
| 饼图 | ❌ | 用数字替代（`2/5 = 40%`） |
| 像素级图形 | ❌ | 需要 Web 仪表盘（P3 #28） |

**定位：TUI 是"快速、轻量、终端内可用"，不是可视化大屏。**

## 五、复杂工作流显示设计

面向未来 D-7 场景：循环迭代、条件分支、嵌套子计划。

### 5.1 循环迭代

```
🔄 task-xxx 性能优化 (迭代模式: 最多 5 次, 目标 P99 < 100ms)
   │
   ├─ 🔄 [迭代 1/5] 优化缓存层    0m45s  结果: P99=120ms ❌ 未达标
   │     ├─ ✅ sub-1a 分析热点     0m12s
   │     ├─ ✅ sub-1b 加 Redis     0m33s
   │     └─ ❌ 验证: P99=120ms    → 继续下一轮
   │
   ├─ 🔄 [迭代 2/5] 优化查询       0m30s  进行中...
   │     ├─ ✅ sub-2a 添加索引     0m18s
   │     └─ 🔄 sub-2b 批量查询     进行中...
   │
   └─ ⏳ [迭代 3/5] 等待前次结果
```

**控制流信息**:

| 元素 | 来源 | 说明 |
|------|------|------|
| 迭代计数器 | `iteration: 2/5` | 当前/最大 |
| 终止条件 | `stop_condition` | `P99 < 100ms` |
| 每次迭代度量 | `iteration_metrics` | `P99=120ms` |
| 是否继续 | 度量 vs 终止条件 | `❌ 未达标 → 继续` |
| 迭代状态聚合 | 子节点汇总 | `[2/5]` 迭代完成数 |

### 5.2 条件分支

```
🔀 task-xxx 方案选择 (条件模式)
   │
   ├─ 🔍 探索阶段 (architect Agent, 只读)
   │     ├─ ✅ sub-1a 调研方案A (Redis)    0m15s
   │     └─ ✅ sub-1b 调研方案B (Memcached) 0m12s
   │
   ├─ 🎯 决策: 选择方案A (Redis)
   │     理由: "Redis 支持持久化和集群，团队有运维经验"
   │
   └─ 📋 执行方案A
         ├─ 🔄 sub-2 实现 Redis 缓存层    进行中...
         └─ ⏳ sub-3 压力测试               等待
```

**控制流信息**:

| 元素 | 来源 | 说明 |
|------|------|------|
| 探索阶段 | `phase: explore` | 🔍 只读分析 |
| 选项列表 | `branches[]` | 方案A / 方案B |
| 决策点 | `decision` | 🎯 选择 + 理由 |
| 已选分支 | `selected_branch` | 正常显示 |
| 未选分支 | 其他 branches | 灰显折叠 |

### 5.3 嵌套子计划（HTN）

```
📋 task-xxx 重构认证模块
   ├─ ✅ sub-1 架构分析              0m30s
   │     agent: architect, 只读
   │
   ├─ 🔄 sub-2 实现 JWT 中间件       1m20s  (子计划: 3 步)
   │     ├─ ✅ sub-2.1 生成 RSA 密钥  0m15s
   │     ├─ ✅ sub-2.2 实现签名逻辑   0m45s
   │     └─ 🔄 sub-2.3 端点暴露       进行中...
   │
   └─ ⏳ sub-3 前端适配                   (子计划: 2 步)
         ├─ ⏳ sub-3.1 登录页改造
         └─ ⏳ sub-3.2 Token 存储
```

**控制流信息**:

| 元素 | 来源 | 说明 |
|------|------|------|
| 嵌套标记 | 父节点 `subtask.has_children` | 显示 `(子计划: N 步)` |
| 层级缩进 | 树结构 depth | `.1 .2 .3` |
| 父节点状态 | 子节点聚合 | 全部完成→父完成 |
| 折叠/展开 | 交互 | Enter 切换，默认折叠 |

### 5.4 复杂度适配策略

| 复杂度 | 显示策略 |
|--------|---------|
| 线性 DAG（当前） | 树形展开，1 层 subtask |
| 循环迭代 | 增加"迭代组"层级，显示度量+终止条件 |
| 条件分支 | 已走路径正常显示，未走路径灰显折叠 |
| HTN 嵌套 | 多层缩进，支持任意深度，默认折叠 |
| 大节点数（>20） | 虚拟滚动 + 全部折叠，Enter 按需展开 |

## 六、验收标准

1. `agent_go status` 默认 TUI 模式

3. 3 tasks × 4 subtasks 同时显示
4. 键盘响应 <100ms
5. 实时日志滞后 <1s
6. 窗口 resize 自动重绘
